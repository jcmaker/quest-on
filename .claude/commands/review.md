# Quest-On Code Review

프로젝트 전체 또는 특정 영역의 코드 건강 상태를 진단한다. PR diff가 아닌 **전체 코드베이스** 기반 리뷰.

---

## 0. Plan Mode 진입 (필수)

**리뷰를 시작하기 전에 반드시 `EnterPlanMode` 도구를 호출하여 plan mode에 진입해라.**

리뷰는 읽기 → 분석 → 보고 작업이다. 코드를 수정하는 것이 아니라 진단하는 것이다. Plan mode는 실수로 파일을 편집하는 것을 방지하고 분석에 집중하게 한다.

리뷰 보고가 완료된 후, 사용자가 수정을 요청하면 그때 `ExitPlanMode`로 나와서 수정을 진행해라.

## 1. 리뷰 범위 결정

인자: `$ARGUMENTS`

| 인자 | 검토 대상 파일 |
|------|---------------|
| (없음) | 전체 코드베이스 — API, auth, grading, chat, submission |
| `auth` | `lib/auth.ts`, `lib/admin-auth.ts`, `proxy.ts`, `lib/get-current-user.ts` |
| `grading` | `lib/grading.ts`, `lib/grading-helpers.ts`, `app/api/session/[sessionId]/grade/` |
| `api` | `app/api/supa/`, `app/api/chat/`, `app/api/ai/` |
| `submission` | `app/api/supa/handlers/submission-handlers.ts`, `app/api/supa/handlers/session-handlers.ts` |

인자가 있으면 해당 영역만 집중 검토. 인자가 없으면 아래 순서대로 전체 검토.

---

## 2. 필수 사전 조사 (반드시 이 순서대로 실행)

아래 체크리스트를 **건너뛰지 말고** 순서대로 수행한 뒤 리뷰를 시작해라.

1. `git log --oneline -15` — 최근 수정 사항 파악. 이미 고친 이슈를 다시 보고하지 마.
2. `lib/auth.ts`, `lib/admin-auth.ts`, `proxy.ts` 읽기 — 인증 플로우 이해
3. `lib/validations.ts` 읽기 — Zod 스키마 + 입력 검증 현황 확인
4. `lib/api-response.ts` 읽기 — 에러 응답 패턴 확인
5. 범위에 해당하는 API 라우트 핸들러 읽기

**각 파일을 실제로 읽어야 한다.** 파일 내용을 추측하지 마.

---

## 3. 프로젝트 컨텍스트 (이미 적용된 보안/안정성 패턴)

리뷰 시 아래 패턴이 이미 존재함을 인지해라. 이것들이 있는데 없다고 보고하면 거짓양성이다.

- **인증**: Clerk `currentUser()` + `unsafeMetadata.role` 기반 역할 확인
- **Admin 인증**: HMAC-signed token 기반 (`lib/admin-auth.ts`)
- **입력 검증**: Zod 스키마 (`lib/validations.ts`)로 API 입력 검증
- **API 응답**: `successJson()` / `errorJson()` 헬퍼 사용 (`lib/api-response.ts`)
- **감사 로그**: `auditLog()` 로 grade/exam/session 이벤트 기록 (`lib/audit.ts`)
- **AI 모델**: dual-model (`AI_MODEL` fast / `AI_MODEL_HEAVY` accuracy), timeout 지원
- **Auth**: `proxy.ts`에서 Clerk middleware 처리 (Next.js 16 convention)

## 4. 심각도 정의 (엄격히 준수)

### P0 — 즉시 수정 필요
프로덕션에서 **지금 당장** 아래 중 하나를 일으킬 수 있는 버그:
- 인증/인가 우회 (권한 없는 사용자가 데이터 접근)
- 데이터 유실 또는 손상 (채점 결과 소실, 세션 데이터 덮어쓰기)
- 서비스 전체 장애 (unhandled exception으로 인한 크래시 루프)

**P0이 아닌 것:**
- 이론적으로만 가능하고 실제 트리거 경로가 없는 취약점
- 에러 메시지가 불친절한 것
- 성능 개선 여지

### P1 — 이번 스프린트에 수정
특정 조건에서 기능이 **의도와 다르게** 동작하는 버그:
- 특정 입력에서 채점이 잘못되는 경우
- race condition으로 데이터 불일치 가능
- 에러 처리 누락으로 사용자에게 500 에러 노출

**P1이 아닌 것:**
- 코드 스타일, 네이밍, 타입 개선
- "더 나은 방법이 있다" 수준의 리팩터링 제안
- 테스트 커버리지 부족

### P2 이하 — 이번 리뷰에서 보고하지 마

## 5. 안티패턴 규칙 (하지 말아야 할 것들)

- **심각도 인플레이션 금지**: "~할 수 있다", "~가 가능하다"만으로 P0/P1 분류 금지. 구체적 트리거 시나리오를 제시해야 함.
- **방어 코드 무시 금지**: Zod 검증, 인증 체크, try-catch 등 이미 존재하는 방어를 확인하지 않고 이슈로 올리지 마. 각 이슈에 대해 기존 방어 코드를 먼저 확인해라.
- **이미 수정된 이슈 보고 금지**: git log에서 확인한 최근 커밋으로 이미 수정된 이슈를 다시 보고하지 마.
- **이슈 수 채우기 금지**: 보고할 만한 이슈가 없으면 없다고 하면 된다. 억지로 찾지 마.
- **UI/프론트엔드 리뷰 금지**: 이 리뷰는 백엔드 로직, API, 인증, 데이터 무결성에 집중한다.
- **중복 보고 금지**: 같은 패턴의 이슈가 여러 파일에 있으면 하나로 묶어서 보고해라.

## 6. 출력 형식

아래 템플릿을 정확히 따라라.

```
## Code Review Report

**리뷰 범위**: [전체 / 특정 영역명]
**리뷰 시각**: [현재 시각]
**최근 커밋**: [git log에서 확인한 최근 3개 커밋 한 줄 요약]

---

### 발견된 이슈

#### [P0] 이슈 제목
- **파일**: `경로:라인번호`
- **트리거 조건**: 이 버그가 발생하는 구체적인 시나리오 (재현 가능한 수준)
- **영향**: 실제로 어떤 일이 발생하는지
- **기존 방어**: 관련 방어 코드가 있다면 명시 → 있는데 우회되는 경로 설명. 없으면 "없음"
- **수정 방안**: 구체적인 코드 변경 제안

---

### 요약
- P0: N건
- P1: N건
- 전체 코드 건강도: [양호 / 주의 / 위험]
```

**이슈가 없는 경우:**
```
## Code Review Report

**리뷰 범위**: [범위]
**최근 커밋**: [요약]

---

### 발견된 이슈

발견된 P0/P1 이슈 없음.

### 요약
- P0: 0건
- P1: 0건
- 전체 코드 건강도: 양호
```

---

## 7. 마무리 규칙

- **P0 최대 3개, P1 최대 5개**. 이 숫자를 채우려고 하지 마.
- **"발견된 이슈 없음"은 유효한 결과다.** 이슈가 없으면 정직하게 보고해라.
- 리뷰 완료 후, 발견한 이슈가 있다면 수정 여부를 사용자에게 물어봐라.
- 리뷰 과정에서 읽은 파일 목록을 내부적으로 추적하되, 최종 출력에는 포함하지 마.
