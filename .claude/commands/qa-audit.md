# QA Code Audit — 소스 코드 감사 v2

테스트가 아닌 **소스 코드 자체**를 읽고, 데이터 흐름을 추적하여 로직 버그를 찾는다.

인자: `$ARGUMENTS` — 감사 대상 (파일 경로, 기능명, 또는 비어있으면 최근 변경 전체)

---

## 핵심 원칙

```
[절대 하지 않을 것]
- 테스트 파일을 읽거나 분석하지 않는다 — 소스 코드만 본다
- "테스트가 통과하므로 문제없다"고 판단하지 않는다
- 코드를 수정하지 않는다 — 발견만 보고한다
- 추측으로 버그를 보고하지 않는다 — 코드에서 확인된 것만 보고

[반드시 할 것]
- 코드가 아니라 **데이터를 따라간다** — "이 데이터를 누가 읽고, 누가 쓰는가?"
- 같은 일을 하는 **형제 코드를 찾아서 비교**한다 — 불일치가 곧 버그다
- **공유 유틸이 있으면, 모든 호출자가 쓰는지** 확인한다 — 안 쓰면 버그다
- "이 값이 null/0/빈배열이면?" 질문을 모든 변수에 던진다
```

---

## Phase 0: 감사 범위 결정 — "데이터를 따라가라"

**파일 중심이 아니라 데이터 중심으로 범위를 잡는다.**

### 0-1. 시드 파일 수집

`$ARGUMENTS`가 있을 때:
- 파일 경로면 → 해당 파일이 시드
- 기능명이면 → Grep으로 관련 파일 검색하여 시드 수집

`$ARGUMENTS`가 비어있으면:
```bash
git diff main...HEAD --name-only -- '*.ts' '*.tsx' ':!__tests__' ':!e2e' ':!*.test.*' ':!*.spec.*'
```

### 0-2. 데이터 맵 구축 (핵심 — 이것이 감사의 기초)

시드 파일에서 **공유 상태**(shared state)를 추출한다:

```
[추출 대상]
1. DB 테이블: .from("테이블명") 패턴 Grep
2. 공유 상태: 전역 변수, context, store, cache
3. API 응답 형식: 클라이언트가 기대하는 데이터 구조
4. 공유 유틸: import해서 쓰는 lib 함수
```

각 공유 상태에 대해 **모든 reader/writer를 수집**한다:

```
[데이터 맵 예시]
grades 테이블:
  Writers: grading.ts (auto), grade/route.ts POST (manual), grading.ts (ai_failed)
  Readers: grade/route.ts GET, final-grades GET, student/report GET, analytics/overview GET
  → 4개 Reader가 교차 비교 대상

sessions.ai_summary:
  Writers: grading.ts (summary), grading-trigger.ts (status)
  Readers: grade/route.ts GET, student/report GET
  → 2개 Reader 비교 + Writer 간 충돌 검사
```

### 0-3. 감사 범위 확정

시드 파일 + 데이터 맵에서 발견된 모든 관련 파일 = 감사 범위.
**시드 파일만 보면 안 된다. 같은 데이터를 건드리는 모든 파일을 읽어야 한다.**

범위가 15개 파일 이상이면 → 데이터 그룹별로 Subagent 병렬 감사 (Phase 6 참조).

## Phase 1: 형제 코드 발견 (Sibling Discovery)

**같은 일을 다른 경로로 수행하는 코드**를 찾는다. 형제 코드 간 불일치는 높은 확률로 버그다.

### 1-1. 형제 유형과 탐색 방법

| 형제 유형 | 탐색 방법 | 예시 |
|-----------|----------|------|
| **트리거 형제** | 같은 함수를 호출하는 N개 진입점 | 3개 엔드포인트가 모두 `triggerGrading()` 호출 |
| **목적 형제** | 같은 결과를 만드는 N개 코드 경로 | 3곳에서 각각 점수를 계산하는데 공식이 다름 |
| **상태 변경 형제** | 같은 데이터를 쓰는 N개 코드 경로 | heartbeat, force-end, feedback 모두 세션을 종료 처리 |
| **구조 형제** | 비슷한 코드 구조를 가진 핸들러 | 비슷한 CRUD 패턴의 API 라우트들 |

### 1-2. 구체적 탐색 전략

```
[트리거 형제 찾기]
1. 시드 파일에서 호출하는 핵심 함수 식별 (예: triggerGrading, calculateScore)
2. Grep으로 같은 함수를 호출하는 다른 파일 전부 수집
3. 각 호출 지점의 전후 로직을 비교

[목적 형제 찾기]
1. 데이터 맵에서 같은 테이블의 Reader 목록 확인
2. 각 Reader가 데이터를 어떻게 가공하는지 비교
3. 계산 공식, 필터 조건, null 처리, 반환 형식의 차이 식별

[상태 변경 형제 찾기]
1. 데이터 맵에서 같은 테이블의 Writer 목록 확인
2. 각 Writer가 어떤 컬럼을 어떤 값으로 설정하는지 비교
3. UPDATE 문의 SET 절을 1:1 대조
```

### 1-3. 비교 체크리스트

형제 코드를 찾았으면 다음을 1:1 대조한다:

```
[형제 코드 비교 항목]
- 같은 컬럼을 SELECT/UPDATE하는가?
- 같은 WHERE 조건을 사용하는가?
- 같은 계산 공식을 사용하는가?
- null/undefined/0/빈배열을 같은 방식으로 처리하는가?
- 에러 처리가 동일한 수준인가?
- 같은 상수(grace period, timeout 등)를 사용하는가?
- 하나에만 있고 다른 쪽에 없는 로직이 있는가?
```

**차이가 발견되면 → 의도적 차이인지, 실수인지 판단한다.** 의도적 차이에는 주석이나 명확한 이유가 있어야 한다. 없으면 버그 후보.

## Phase 2: 불변량 발견 (Invariant Discovery)

**코드베이스에서 "이 규칙이 깨지면 버그"인 불변량을 스스로 발견**한다.

### 2-1. 공유 유틸 불변량

```
[탐색]
1. lib/ 또는 utils/ 에서 export된 함수 목록 추출
2. 각 함수의 이름과 목적 파악 (예: calculateOverallScore, deduplicateGrades)
3. 같은 목적의 인라인 코드가 다른 파일에 있는지 Grep

[불변량]
"공유 유틸 X가 존재하면, X의 목적과 같은 작업을 하는 모든 코드는 X를 사용해야 한다"
위반 = 같은 작업을 인라인으로 다시 구현한 코드
```

### 2-2. 패턴 전파 불변량

```
[탐색]
1. 같은 역할의 파일들(예: API route handlers)을 수집
2. 다수가 따르는 공통 패턴 식별 (예: lazy Supabase getter, rate limiting)
3. 해당 패턴을 따르지 않는 소수 파일 식별

[불변량]
"같은 역할의 N개 파일 중 대다수가 패턴 P를 따르면, 나머지도 따라야 한다"
위반 = 다수와 다른 방식으로 초기화/처리하는 소수 파일
```

### 2-3. 데이터 형식 불변량

```
[탐색]
1. Writer가 저장하는 데이터의 형식/구조 확인
2. Reader가 기대하는 데이터의 형식/구조 확인
3. Writer-Reader 간 형식 불일치 검사

[불변량]
"같은 데이터를 쓰는 모든 Writer는 같은 형식으로 저장해야 한다"
위반 = Writer A는 {text: "..."}, Writer B는 "..." (형식 불일치)
```

### 2-4. 상수 일관성 불변량

```
[탐색]
1. 같은 개념의 상수가 여러 파일에 정의되어 있는지 검색
   (예: grace period, timeout, max retries)
2. 값이 동일한지 비교

[불변량]
"같은 개념의 상수는 같은 값이어야 한다 (또는 하나의 정의를 공유해야 한다)"
위반 = 파일 A는 GRACE_PERIOD=5s, 파일 B는 gracePeriod=30s
```

## Phase 3: 실행 경로 추적 (Path Tracing)

각 엔드포인트(또는 핵심 함수)에 대해 데이터가 흘러가는 전체 경로를 추적한다.

### 3-1. 정상 경로 (Happy Path)

요청 → 인증 → 파라미터 파싱 → 비즈니스 로직 → DB 쿼리 → 응답 변환 → 반환

각 단계 경계에서 확인:
- 이전 단계의 출력 타입이 다음 단계의 입력 기대와 일치하는가?
- DB SELECT에 이후 로직에서 사용하는 컬럼이 모두 포함되는가?
- WHERE 조건이 의도한 데이터만 필터링하는가?
- 조건 체크에서 truthiness(`if (value)`)와 null체크(`if (value != null)`)를 구분하는가?
  - `0`, `""`, `false`가 유효한 값인 변수에 truthiness를 쓰면 버그

### 3-2. 에러 경로 (Error Paths)

각 `if/else`, `try/catch`, early return을 나열하고:
- 에러 시 부분적으로 완료된 작업의 정합성이 유지되는가?
- 에러 응답 형식이 클라이언트의 기대와 일치하는가?
- catch에서 에러를 삼키고(swallow) 잘못된 기본값을 반환하지 않는가?

---

## Phase 4: 엣지케이스 열거 (Edge Case Enumeration)

### 4-1. 값 기반 — 모든 입력 변수에 적용

| 타입 | 엣지케이스 | 특히 확인할 것 |
|------|-----------|--------------|
| 배열 | `[]`, 단일 요소 | `reduce()` 초기값 없이 빈 배열, `.length` 나눗셈 분모 |
| 숫자 | `0`, 음수, `NaN` | `if (score)` → 0이 falsy, 나눗셈 분모 |
| 문자열 | `""`, `null`, `undefined` | `if (name)` → ""이 falsy |
| 객체 | `{}`, 필수 키 누락 | optional chaining 후 기본값 |

### 4-2. 시나리오 기반 — 비즈니스 로직에 적용

| 시나리오 | 확인 사항 |
|----------|----------|
| 전부 실패 | 모든 항목 실패 → 결과 배열 빈 → 후속 계산 가능? |
| 부분 실패 | N개 중 일부 실패 → 성공분만으로 정합적 결과? |
| 중복 입력 | 같은 key 2개 → upsert? 에러? 무시? |
| 빈 입력 | 데이터 0건 → divide by zero, null access? |

## Phase 5: 동시성 검사 (Concurrency Audit)

### 5-1. Read-Modify-Write 탐지

```
[패턴 검색]
1. SELECT/fetch → 가공 → UPDATE/insert 순서의 코드 찾기
2. 두 요청이 동시에 실행되면 하나의 쓰기가 사라지는가?
3. CAS, 트랜잭션, 낙관적 잠금 중 하나로 보호되는가?
```

### 5-2. 멱등성 검사

같은 요청 2번 → 결과가 동일한가? 데이터가 중복 생성되는가?

### 5-3. 부분 실패 검사

`Promise.all()` 또는 배치 작업에서 일부 실패 시:
- 성공한 작업의 결과가 보존되는가?
- 실패한 작업이 재시도 가능한가?

## Phase 6: 병렬 감사 오케스트레이션

감사는 **2단계 fan-out/fan-in** 구조로 실행한다. Agent Teams는 불필요 — 에이전트 간 대화 없이 오케스트레이터(메인 세션)가 조율한다.

### 6-1. Stage A — 병렬 (데이터 맵 + 불변량)

두 Subagent를 **동시에** 디스패치:

```
Agent A-1: 데이터 맵 구축
  - .from("테이블명") Grep으로 전체 reader/writer 수집
  - 테이블별 Writer/Reader 목록 반환
  - 중복 상수 (grace period, timeout 등) 식별
  → 결과: 데이터 맵 JSON

Agent A-2: 불변량 발견 (데이터 맵과 독립적)
  - 공유 유틸 채택 여부 전수 검사
  - 패턴 전파 (lazy getter, rate limit, UUID validation, error logging) 검사
  - 에러 상세 노출 검사
  → 결과: 불변량 위반 목록
```

**불변량 발견은 데이터 맵이 필요 없다** — 파일 패턴 검사이므로 독립 실행 가능.

### 6-2. Stage B — 데이터 맵 기반 형제 발견 (병렬)

Stage A의 데이터 맵 결과를 받아서, **데이터 그룹별로** 형제 발견 Subagent를 디스패치:

```
[오케스트레이터가 하는 일]
1. 데이터 맵에서 테이블별 Writer/Reader 목록 추출
2. 테이블을 의미적으로 그룹화 (예: sessions+exams, submissions+messages, grades)
3. 각 그룹에 대해 Subagent 디스패치

Agent B-1: sessions/exams 그룹 형제 비교
  입력: sessions Writer 17개 + Reader 27개 목록, exams Writer 6개 + Reader 30+개 목록
  작업: 제출 경로 5개 비교, 시간 계산 4개 비교, 상태 전이 비교
  → 결과: 형제 비교 발견 목록

Agent B-2: submissions/messages 그룹 형제 비교
  입력: submissions Writer 6개 + Reader 13개 목록, messages Writer 2개 + Reader 13개 목록
  작업: 압축 데이터 형식 비교, 질문 정규화 비교, 데이터 소스 (client vs server) 비교
  → 결과: 형제 비교 발견 목록

Agent B-3: grades 그룹 형제 비교
  입력: grades Writer 3개 + Reader 9개 목록
  작업: 점수 계산 공식 비교, grade_type 처리 비교, deduplication 채택 비교
  → 결과: 형제 비교 발견 목록
```

각 Agent에게 **데이터 맵에서 해당 그룹의 모든 파일 경로**를 전달한다.
이렇게 하면 에이전트가 검색할 필요 없이 **바로 비교에 집중**할 수 있다.

### 6-3. 결과 통합

```
[오케스트레이터가 하는 일]
1. Stage A-2 (불변량) + Stage B-* (형제 비교) 결과 취합
2. 중복 발견 제거 (같은 파일:줄번호가 다른 기법으로 잡힌 경우)
3. 교차 그룹 분석: B-1의 sessions 발견이 B-3의 grades에 영향을 주는지
4. 심각도 최종 판정
5. 단일 리포트 생성
```

### 6-4. 소규모 감사 시 (15개 파일 미만)

Subagent 없이 메인 세션에서 순차 실행:
1. 데이터 맵 구축 (Phase 0)
2. 형제 발견 (Phase 1) — 데이터 맵 기반
3. 불변량 발견 (Phase 2)
4. 경로 추적 + 엣지케이스 + 동시성 (Phase 3-5)
5. 보고 (Phase 7)

## Phase 7: 발견 사항 보고

### 심각도 분류

| 심각도 | 기준 | 판단 근거 |
|--------|------|----------|
| **CRITICAL** | 데이터 손실, 잘못된 결과 노출 | 코드에서 확인된 구체적 입력 → 구체적 잘못된 출력 |
| **HIGH** | 특정 시나리오에서 오작동 | 형제 코드 비교에서 누락된 로직, 불변량 위반 |
| **MEDIUM** | 일관성 깨짐 | 패턴 미전파, 상수 불일치, 형식 불일치 |
| **LOW** | 방어 코드 부재 | 현재는 발생 안 하지만 코드 변경 시 깨질 수 있음 |

### 거짓 양성 방지

보고 전 반드시 확인:
- **"이 차이가 의도적일 수 있는가?"** — 주석, 변수명, 커밋 메시지에 이유가 있으면 보고하지 않는다
- **"실제로 이 경로가 실행될 수 있는가?"** — dead code면 보고하지 않는다
- **"이미 다른 레이어에서 방어되고 있는가?"** — 호출자가 이미 검증하면 중복 검사 불필요

### 출력 형식

## 8. 규칙

- **소스 코드만 읽는다**. 테스트 파일(`__tests__/`, `e2e/`, `*.test.*`, `*.spec.*`)은 감사 대상이 아니다.
- **코드를 수정하지 않는다**. 발견 사항만 보고한다.
- **데이터를 따라간다**. 시드 파일만 보지 않는다. 같은 데이터를 건드리는 모든 파일을 읽는다.
- **형제를 찾는다**. 같은 일을 하는 코드가 2개 이상이면 반드시 비교한다.
- **불변량을 발견한다**. 공유 유틸, 공통 패턴, 상수가 있으면 전체 채택 여부를 확인한다.
- 확인된 사실만 보고한다. "~일 수 있다"가 아니라 "이 코드는 X를 하지만 Y가 빠졌다"로 쓴다.
- 발견 사항이 없으면 "감사 완료, 이상 없음"이라고 보고한다. 억지로 문제를 찾지 않는다.
- 감사 완료 후 CRITICAL/HIGH 발견 시 수정 여부를 사용자에게 물어본다.
