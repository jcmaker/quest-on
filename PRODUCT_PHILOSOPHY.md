# PRODUCT_PHILOSOPHY.md

## 1. 우리는 "만드는 것"보다 "쓸 이유"를 먼저 증명한다

좋은 프로덕트는 기능이 많은 앱이 아니다.

좋은 프로덕트는 사용자가 특정 순간에 자연스럽게 떠올리고, 다시 돌아오고, 굳이 설명하지 않아도 쓸 수 있는 도구다.

따라서 개발자는 항상 아래 질문에서 시작한다.

- 이 기능은 사용자의 어떤 불편을 줄이는가?
- 사용자는 이 기능을 언제, 어디서, 왜 쓰는가?
- 이 기능이 없어도 제품의 핵심 경험이 유지되는가?
- 이 기능을 추가하면 제품이 더 명확해지는가, 더 복잡해지는가?

기능은 만들 수 있다고 만드는 것이 아니다.
사용자의 행동을 더 쉽게 만들 때만 만든다.

---

## 2. MVP는 "대충 만든 버전"이 아니라 "핵심만 남긴 버전"이다

MVP는 미완성품이 아니다.

MVP는 제품의 본질을 검증하기 위해 불필요한 것을 제거한 가장 작은 실험이다.

MVP에 들어가야 하는 것은 다음 세 가지뿐이다.

1. 사용자가 처음 들어왔을 때 바로 이해할 수 있는 핵심 화면
2. 사용자가 실제로 가치를 느끼는 핵심 행동
3. 다시 돌아올 이유를 만드는 최소한의 저장/반복 구조

반대로 다음은 MVP에서 조심해야 한다.

- 과한 설정 화면
- 처음부터 완벽한 디자인 시스템
- 복잡한 권한 구조
- 커뮤니티 기능
- 관리자 기능
- 고급 필터
- 지나치게 많은 온보딩
- "나중에 필요할 것 같은" 추상화

MVP의 목적은 완성도가 아니라 검증이다.
하지만 "대충"과 "작게"는 다르다.
작게 만들더라도 핵심 경험은 제대로 작동해야 한다.

---

## 3. 기능은 추가보다 제거가 어렵다

기능을 추가하는 것은 쉽다.
기능을 제거하는 것은 어렵다.

한 번 들어간 기능은 사용자 경험, 데이터 구조, UI, 코드베이스, 문서, 테스트, 유지보수 비용에 모두 영향을 준다.

그래서 기능을 추가하기 전에 반드시 생각한다.

- 이 기능은 핵심 흐름을 방해하지 않는가?
- 이 기능 때문에 사용자가 고민해야 할 선택지가 늘어나지 않는가?
- 이 기능 없이 더 단순한 방식으로 해결할 수 없는가?
- 이 기능을 나중에 제거하기 쉽게 만들 수 있는가?

좋은 제품은 많은 기능을 가진 제품이 아니라,
필요 없는 기능이 적은 제품이다.

---

## 4. 사용자는 설명서를 읽지 않는다

사용자는 우리가 생각한 흐름대로 움직이지 않는다.
사용자는 UI를 훑고, 대충 누르고, 안 되면 나간다.

따라서 좋은 UI는 설명이 아니라 구조로 이해되어야 한다.

원칙은 다음과 같다.

- 중요한 행동은 눈에 보여야 한다.
- 사용자가 다음에 뭘 해야 할지 고민하지 않아야 한다.
- 빈 화면도 기능의 일부다.
- 에러 메시지는 개발자 로그가 아니라 사용자 안내문이어야 한다.
- 설정 없이도 기본값만으로 쓸 수 있어야 한다.
- 사용자가 실수해도 되돌릴 수 있어야 한다.

좋은 UX는 친절한 문구보다 좋은 기본 흐름에서 나온다.

---

## 5. 데이터 구조는 제품의 뼈대다

UI는 바꿀 수 있다.
디자인도 바꿀 수 있다.
하지만 잘못 잡은 데이터 구조는 오래 남는다.

데이터 모델을 설계할 때는 지금 화면만 보지 않는다.
사용자의 실제 행동과 미래의 확장 가능성을 같이 본다.

좋은 데이터 구조는 다음 조건을 만족한다.

- 사용자의 행동 단위와 잘 맞는다.
- 불필요하게 중첩되지 않는다.
- 동기화와 오프라인 사용을 고려한다.
- 나중에 검색, 필터, 정렬하기 쉽다.
- 삭제, 수정, 복구 흐름을 고려한다.
- 서버와 클라이언트의 책임이 분명하다.

데이터 구조가 복잡해지는 순간, 제품도 복잡해진다.

---

## 6. 코드는 제품 속도를 결정한다

나쁜 코드는 오늘은 빠르게 보이지만, 내일의 속도를 빼앗는다.

좋은 코드는 멋진 코드가 아니다.
좋은 코드는 다음 사람이 읽고, 수정하고, 삭제할 수 있는 코드다.

우리는 다음을 지킨다.

- 코드는 작게 나눈다.
- 이름은 길어져도 명확하게 짓는다.
- 숨겨진 부작용을 줄인다.
- 중복보다 성급한 추상화를 더 경계한다.
- 타입으로 의도를 표현한다.
- 에러 처리를 흐름의 일부로 본다.
- UI, 상태, 비즈니스 로직을 가능한 분리한다.
- 파일 하나가 너무 많은 책임을 갖지 않게 한다.

코드는 컴퓨터에게 명령하는 글이지만,
실제로 오래 읽는 사람은 인간이다.

---

## 7. 추상화는 세 번째 반복부터 한다

처음부터 완벽한 구조를 만들려고 하지 않는다.

처음에는 명확하게 만든다.
두 번째에는 중복을 관찰한다.
세 번째에 패턴이 보이면 추상화한다.

성급한 추상화는 코드를 유연하게 만드는 것이 아니라,
아직 모르는 문제를 아는 척하는 것이다.

추상화하기 전에 묻는다.

- 정말 같은 문제인가?
- 앞으로도 같은 방식으로 변할 것인가?
- 추상화 때문에 읽기 어려워지지 않는가?
- 그냥 중복을 두는 편이 더 안전하지 않은가?

중복은 비용이다.
하지만 잘못된 추상화는 부채다.

---

## 8. 기술 선택은 자랑이 아니라 책임이다

새로운 기술을 쓰는 이유는 "멋있어서"가 아니다.
문제를 더 잘 해결하기 위해서다.

기술을 선택할 때는 다음을 본다.

- 이 기술이 현재 문제를 단순하게 만드는가?
- 팀 또는 개인이 유지보수할 수 있는가?
- 문서와 커뮤니티가 충분한가?
- 장애가 났을 때 디버깅 가능한가?
- 기존 스택과 충돌하지 않는가?
- 6개월 뒤에도 이 선택을 이해할 수 있는가?

기술은 제품을 돕는 도구다.
기술 자체가 제품의 주인공이 되면 안 된다.

---

## 9. AI는 개발자를 대체하는 것이 아니라 사고를 증폭한다

AI에게 모든 것을 맡기지 않는다.
AI는 빠르게 초안을 만들고, 실수를 찾고, 반복 작업을 줄이는 도구다.

하지만 최종 책임은 개발자에게 있다.

AI를 사용할 때는 다음을 지킨다.

- AI가 만든 코드는 반드시 읽는다.
- 모르는 코드는 그대로 붙여넣지 않는다.
- 큰 변경은 작은 단계로 나눈다.
- 생성된 코드가 기존 구조와 맞는지 확인한다.
- 타입 에러, 런타임 에러, 보안 문제를 직접 검토한다.
- AI가 만든 설명보다 실제 코드와 실행 결과를 믿는다.

AI는 속도를 높여준다.
하지만 방향을 정하는 것은 개발자다.

---

## 10. 제품의 첫 화면은 약속이다

사용자가 처음 보는 화면은 제품이 사용자에게 하는 약속이다.

첫 화면은 다음을 말해줘야 한다.

- 이 앱이 무엇을 하는지
- 사용자가 무엇을 할 수 있는지
- 지금 당장 무엇을 누르면 되는지
- 이 앱을 쓰면 어떤 점이 편해지는지

첫 화면에서 사용자를 설득하지 못하면,
뒤에 있는 좋은 기능은 아무 의미가 없다.

---

## 11. 저장되는 순간 제품은 진지해진다

사용자가 데이터를 입력하고 저장하는 순간,
그 제품은 단순한 화면이 아니라 사용자의 기억을 맡는 도구가 된다.

그래서 저장 기능은 가볍게 다루면 안 된다.

반드시 고려한다.

- 저장 실패 시 사용자는 알 수 있는가?
- 임시 저장이 필요한가?
- 삭제한 데이터를 복구할 수 있는가?
- 오프라인 상태에서는 어떻게 되는가?
- 동기화 충돌은 어떻게 처리하는가?
- 사용자가 데이터를 내보낼 수 있는가?
- 계정 없이 만든 데이터는 나중에 계정과 연결할 수 있는가?

사용자의 데이터는 제품의 가장 중요한 자산이다.
그리고 그 자산은 개발자의 것이 아니다.

---

## 12. 좋은 제품은 사용자를 똑똑하게 만들지 않는다

좋은 제품은 사용자가 많이 배워야 쓸 수 있는 제품이 아니다.

좋은 제품은 사용자가 원래 하던 행동을 더 편하게 하도록 돕는다.

예를 들어 사용자가 이미 메모장에 장소를 저장하고 있다면,
처음부터 복잡한 지도 앱을 강요하지 않는다.

먼저 익숙한 메모 경험을 제공하고,
그 안에서 장소, 이미지, 지도, 태그, 일정 같은 기능을 자연스럽게 확장한다.

제품은 사용자를 바꾸려 들기보다,
사용자의 기존 습관 위에 올라타야 한다.

---

## 13. 성능은 나중에 하는 일이 아니다

성능 최적화는 마지막 단계에서만 하는 일이 아니다.

특히 모바일 앱과 웹앱에서는 첫 로딩, 입력 반응성, 스크롤 부드러움, 이미지 로딩이 곧 사용자 경험이다.

기본 원칙은 다음과 같다.

- 첫 화면에 필요 없는 코드는 늦게 불러온다.
- 큰 이미지는 최적화한다.
- 리스트는 가상화나 페이지네이션을 고려한다.
- 입력 중 불필요한 렌더링을 줄인다.
- 서버 요청은 캐싱과 실패 처리를 고려한다.
- 느린 작업에는 로딩 상태를 보여준다.

빠른 제품은 신뢰감을 준다.
느린 제품은 사용자를 불안하게 만든다.

---

## 14. 보안은 기능이 아니라 기본값이다

보안은 나중에 붙이는 기능이 아니다.
처음부터 기본값이어야 한다.

반드시 지킨다.

- 비밀키는 클라이언트에 노출하지 않는다.
- 사용자 입력은 검증한다.
- 인증과 권한을 구분한다.
- 서버에서 다시 권한을 확인한다.
- 민감한 데이터는 최소한만 저장한다.
- 로그에 개인정보를 남기지 않는다.
- 외부 API 응답을 그대로 신뢰하지 않는다.

"아직 MVP니까 괜찮다"는 말은 보안 문제에 적용되지 않는다.

---

## 15. 에러는 실패가 아니라 제품 경험의 일부다

에러는 반드시 발생한다.

네트워크는 끊기고, 서버는 실패하고, 사용자는 이상한 값을 입력하고, 외부 API는 예상과 다른 응답을 준다.

좋은 제품은 에러가 없는 제품이 아니라,
에러가 났을 때 사용자를 버리지 않는 제품이다.

에러 처리 원칙은 다음과 같다.

- 사용자가 이해할 수 있는 메시지를 보여준다.
- 다시 시도할 방법을 제공한다.
- 입력한 내용을 잃지 않게 한다.
- 개발자가 디버깅할 수 있는 로그를 남긴다.
- 조용히 실패하지 않는다.

사용자가 가장 불안한 순간에 제품의 신뢰도가 결정된다.

---

## 16. 개발자는 미래의 자신에게 친절해야 한다

지금 급하게 만든 코드는 미래의 내가 고친다.

그래서 오늘의 나는 미래의 나에게 친절해야 한다.

- 왜 이렇게 만들었는지 남긴다.
- 애매한 로직은 주석으로 의도를 설명한다.
- TODO는 구체적으로 쓴다.
- 임시 코드는 임시라고 표시한다.
- 삭제 가능한 코드는 삭제하기 쉽게 둔다.
- 복잡한 결정은 문서로 남긴다.

코드는 현재의 문제 해결이면서,
미래의 유지보수자에게 보내는 편지다.

---

## 17. 완벽한 설계보다 빠른 피드백이 낫다

혼자 오래 고민해서 만든 완벽한 설계보다,
작게 만들어 실제로 써보고 얻은 피드백이 더 강하다.

제품 개발은 추측을 줄이는 과정이다.

우리는 다음 순서를 따른다.

1. 작게 만든다.
2. 직접 써본다.
3. 불편한 지점을 찾는다.
4. 고친다.
5. 다시 반복한다.

좋은 제품은 한 번에 나오지 않는다.
좋은 제품은 반복하면서 선명해진다.

---

## 18. 삭제하기 쉬운 코드를 만든다

좋은 코드는 추가하기 쉬운 코드만이 아니다.
좋은 코드는 삭제하기 쉬운 코드다.

삭제하기 쉬운 코드는 다음 특징을 가진다.

- 책임이 분리되어 있다.
- 사이드 이펙트가 적다.
- 의존성이 명확하다.
- 한 기능이 여러 곳에 흩어져 있지 않다.
- 데이터 흐름이 추적 가능하다.

언젠가 바뀔 것을 전제로 만들면,
오늘의 코드도 더 단단해진다.

---

## 19. 제품의 방향은 기능 목록이 아니라 사용자 시나리오로 정한다

기능 목록만 보면 제품은 점점 복잡해진다.

대신 사용자 시나리오로 생각한다.

나쁜 질문:

- 지도 기능 넣을까?
- 태그 기능 넣을까?
- 커뮤니티 넣을까?
- AI 추천 넣을까?

좋은 질문:

- 사용자가 인스타에서 본 카페를 나중에 잊지 않으려면 어떤 흐름이 필요한가?
- 사용자가 저장한 장소를 다시 찾을 때 무엇이 가장 먼저 떠오르는가?
- 사용자가 여행 하루 전날 앱을 열면 무엇을 보고 싶어 하는가?
- 사용자가 친구에게 장소 리스트를 공유하려면 어떤 형태가 가장 편한가?

기능은 시나리오에서 나온다.
시나리오 없이 나온 기능은 제품을 흐린다.

## 20. 최종 원칙

우리는 다음을 지킨다.

- 작게 만든다.
- 명확하게 만든다.
- 사용자가 이해할 수 있게 만든다.
- 데이터를 소중히 다룬다.
- 코드는 읽기 쉽게 만든다.
- 기술보다 문제를 먼저 본다.
- AI를 쓰되 책임은 개발자가 진다.
- 빠르게 검증하고, 천천히 복잡하게 만든다.
- 기능을 늘리는 것보다 핵심 경험을 선명하게 만든다.

좋은 프로덕트는 개발자의 욕심을 보여주는 것이 아니다.

좋은 프로덕트는 사용자의 삶에서 작지만 확실한 불편 하나를 조용히 없애는 것이다.
